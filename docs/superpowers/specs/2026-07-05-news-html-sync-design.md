# news.html 継続更新の仕組み化 設計書

## 背景・目的

`report/` フォルダに今後も調査レポートが追加され続ける。追加のたびに、その内容を
`history/news.html`（直近トレンドをまとめた手書きHTML）へ反映する運用を続けていくため、
毎回同じ判断基準・手順で更新できるよう Claude Code のスキルとして仕組み化する。

## スコープ

- 新スキル `.claude/skills/sync-news-html/` を追加し、`report/` の1レポートを
  `history/news.html` に反映する作業を標準化する。
- カード増加によるページ肥大化を防ぐため、循環先として `history/archive.html` を新設する。
- `format-report` スキルと同様、Claude Code から手動で呼び出す運用とする（自動トリガーは対象外）。

## 発火条件・入力

- ユーザーが `report/XXX.md をnews.htmlに反映して` のように対象ファイルを明示して呼び出す。
- 指定がなければ `report/` 内で最終更新日時が最も新しいファイルを対象にする。
- 対象レポートが未整形（改行のない塊）であれば、抽出前にまず `format-report` スキルを適用する。

## 抽出・分類ロジック

- レポート全文を読み、見出しになりうるニューストピック（固有名詞・数字を含むもの）を候補として洗い出す。
- 各候補は既存7テーマ（エージェント/国内実装/フィジカルAI/ソブリンAI/インフラ/経済社会/セキュリティ）の
  うち最も近いものに割り当てる。新テーマは追加しない。
- 重複判定: 既存カードと同一の企業・イベントを指す内容があれば、新規カードは作らず
  当該カードの `.src` 行に今回のレポートのファイル名を追記するだけにとどめる。

## カード上限・循環・archive.html

- 各セクションのカード上限は **6枚**（現状の分布に合わせる）。
- 上限を超える場合、そのセクションで最も古いカードを `history/news.html` から取り除き、
  `history/archive.html` の対応テーマセクションへ移す。
- カードには非表示のメタデータを付与する:
  - `data-added="YYYY-MM-DD"` — 追加日（新旧判定・循環対象の特定に使用）
  - `data-report="ファイル名"` — 出典レポート（重複判定に使用）
- `history/archive.html` は `news.html` と同じ7テーマ構成・同じカードデザイン(CSS再利用)を
  持つ「過去ログ」ページとして新規作成する。ticker・hero・quick-filtersは不要、
  シンプルなヘッダーとセクション一覧のみで構成する。
- ナビゲーション導線（今回の追加要件）:
  - `news.html` の `nav-links` に `archive.html` への恒常リンクを追加する。
  - `news.html` の `cta-back` に「過去ログを見る →」リンクを追加する。
  - `archive.html` 側にも `news.html` に戻るリンクを設置し、双方向にする。

## 統計値・ティッカーの同期

- hero文中の「調査レポート15本を横断」、footerの「COMPILED FROM 15 RESEARCH REPORTS」は、
  更新のたびに `report/` 配下の実ファイル数を数えて置き換える。
- 上部のBREAKINGティッカーは新規カード追加時に見出しを1〜2件追記し、
  あふれた分は古いものから削除して総件数をおおむね8件程度に維持する。

## 変更報告・レビューフロー

- `news.html` / `archive.html` はスキルが直接上書き保存する（git管理下のため差分は追跡可能）。
- 作業完了後、以下をユーザーに要約報告する:
  - 追加したカード（セクション名・見出し）
  - 循環してarchiveに移したカード
  - 更新した統計値（レポート本数、ティッカー内容）
- ユーザーは報告を見て問題があれば個別に修正を依頼する運用とする（コミット前の逐一承認は不要）。

## 対象外（将来の拡張候補）

- ニュース項目をJSON等でデータ層として持ち、テンプレートから静的生成する構成
  （カード数がさらに増えた場合の移行候補として、今回は採用しない）。
- レポート追加の自動検知・自動反映（現時点は手動トリガーのみ）。
